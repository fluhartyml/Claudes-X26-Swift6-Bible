Reference/ContentView

  1. ContentView (ContentView.swift)

  2. Reference: SwiftUI View convention

  3. What this file is

  4. ContentView is the SwiftUI view Xcode creates as the starting point for every new project. Its name is convention only — nothing in SwiftUI requires it. Apple uses "ContentView" because it's neutral; you'd rename it (or split it into many smaller views) for a real app.

  5. import SwiftUI

struct ContentView: View {
    var body: some View {
        VStack {
            Image(systemName: "globe")
                .imageScale(.large)
                .foregroundStyle(.tint)
            Text("Hello, world!")
        }
        .padding()
    }
}

#Preview {
    ContentView()
}

  6. The View protocol

  7. Every SwiftUI view is a struct conforming to View. The protocol has one requirement: a body property of some View. Whatever you return from body is what gets rendered. The struct is a description of the UI, not a reference to a UIKit/AppKit view object — SwiftUI builds the actual view tree from your description on each redraw.

  8. Why some View and not just View

  9. some View is an opaque return type — the compiler knows the concrete type, but callers don't. SwiftUI's view types are deeply nested generic structs (VStack<TupleView<(Image, Text)>> and so on). Spelling them out explicitly is impossible to read; some View hides the noise. The compiler still type-checks everything; you just don't have to type the full type yourself.

  10. The #Preview macro

  11. #Preview tells Xcode's canvas what to render in the preview pane. You can have multiple #Preview blocks in the same file with different setups (different data, different size class, different appearance). Previews don't ship in the built app — they're a development-time convenience.

  12. Composing views

  13. You build complex UI by nesting smaller views. Best practice is to keep each view small — under 50 lines — and pull subtrees into their own struct ChildView: View when they grow. Compiler errors get clearer, previews render faster, and the views become reusable.

  14. Rule of thumb: if your body needs scrollbars in the editor, it's too big. Extract subviews.

  15. State, bindings, and the rest of the view-data world

  16. ContentView starts as a stateless view. As it grows you'll add:

    • @State — private mutable state owned by the view.
    • @Binding — a two-way connection to state owned higher up the tree.
    • @Environment — system-provided values (color scheme, dynamic type size, scene phase).
    • @Bindable — for binding into @Observable reference types.

  18. View identity

  19. SwiftUI reconstructs your view tree on every state change. To remember "this view here is the same view from before" (so animations work, state persists), it uses view identity — either the structural position in the tree or an explicit .id() modifier. Mismatched identity is the source of most "my animation jumped" bugs.
