Reference/DeveloperNotes.swift

  1. DeveloperNotes.swift (the project memory file)

  2. Reference: Michael's project convention

  3. What it is

  4. Every Xcode project Michael builds gets a file at the top of its source tree named after the project: MyProject_DeveloperNotes.swift. The file is a regular Swift file as far as Xcode is concerned — it lives in the project, the filesystem-synced project picks it up, the navigator shows it — but the entire file is wrapped in // line comments. The compiler reads the comments, throws them away, and emits zero bytes for the file. Nothing about DeveloperNotes.swift ships in the built app.

  5. //
//  TngrnGrvWr_DeveloperNotes.swift
//  TngrnGrvWr (Tangerine GrooveWire)
//
//  Developer Notes — Persistent Memory for AI Assistants
//

// ============================================================================
// MARK: - PROJECT IDENTITY
// ============================================================================
//
//  Name:           Tangerine GrooveWire
//  Bundle ID:      com.TangerineGrooveWire.TngrnGrvWr
//  Platform:       iOS / macOS (Universal)
//  Version:        1.0
//  Storage:        SwiftData + CloudKit
//  Deployment:     iOS 18.6 / macOS 15.6
//
//  Code:           [empty — the whole file is comments]

  6. Why a Swift file and not a Markdown / text file?

  7. Two reasons it lives as a .swift rather than NOTES.md or README.md:

    • Xcode shows it in the navigator alongside the code. A Markdown file in the project root would also show up, but Swift files cluster with the rest of the source tree visually. Anyone opening the project cold sees MyProject_DeveloperNotes.swift right next to MyProjectApp.swift and clicks it first.
    • Xcode's MARK pragma works in comments. When you write // MARK: - PROJECT IDENTITY, Xcode's symbol jump bar (Ctrl+6 or click the breadcrumb at the top of the editor) shows each MARK as a navigable section. You can jump straight to "App Store Connect notes" or "Architecture decisions" in a long file. A markdown file doesn't get that treatment.

  9. What goes in it

  10. Anything a future reader (you, a collaborator, an AI session) would want to know without grep-walking the codebase. Common sections:

    • PROJECT IDENTITY — name, bundle ID, platforms, version, GitHub URL, deployment target, storage stack.
    • APP STORE CONNECT — current submission status, App Store ID, common rejection notes.
    • ARCHITECTURE DECISIONS — "we use SwiftData not Core Data because..." / "view models are @Observable, not ObservableObject." Decisions worth documenting so they don't get re-litigated.
    • OPEN QUESTIONS — things to investigate next, parked work, "this only works because of X but it's fragile."
    • VERSION HISTORY — each shipped version's changelog at the source level (which is more honest than App Store "what's new" copy).

  12. Sync to the GitHub wiki

  13. Per Michael's convention (feedback_wiki_sync.md in the apartment memory), DeveloperNotes.swift for any project gets mirrored to a Developer-Notes.md page on that project's GitHub wiki. The Swift file is the source of truth; the wiki page is generated by stripping the // prefixes from each line. Readers browsing the repo on GitHub see the same content.

  14. The wiki sync isn't automated — it's done manually after each meaningful update. Don't let them drift; if the Swift file is ahead by a major version of notes, regenerate the wiki page.

  15. The "compiler skips it" trick — why this works

  16. Swift's compiler treats // as line comment all the way to end-of-line. A file that's nothing but comment lines is syntactically valid — it parses to "an empty file" and emits no symbols. The build succeeds. The output binary doesn't grow. The file just exists for human readers.

  17. You can also use /* ... */ block comments to wrap structured content, but line comments win because they keep MARK pragmas active and don't risk an unbalanced */ ending the comment unexpectedly.

  18. Watch out for

  19. Don't accidentally uncomment. If you copy a code snippet into the Swift file to remind yourself of an approach, comment every line. An uncommented import Foundation won't break anything, but uncommented function declarations or top-level expressions will conflict with the rest of your project and you'll spend an hour debugging "why is this symbol declared twice?"
  20. Don't put secrets in it. The file lives in your repo. If your repo is public, your "developer notes" are public too. API keys, tokens, passwords, internal URLs — out. Use the apartment secrets.local.md pattern (gitignored) for those.
