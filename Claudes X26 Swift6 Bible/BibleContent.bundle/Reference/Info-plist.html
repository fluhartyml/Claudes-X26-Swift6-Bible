Reference/Info.plist

  1. Info.plist

  2. Reference: bundle metadata file

  3. What this file is

  4. Info.plist is a property list — an Apple-flavored XML or binary plist — that ships inside every app bundle and tells the operating system everything it needs to know about your app before launching it. Bundle ID, version, supported orientations, required device capabilities, what privacy permissions you need: it all lives here.

  5. Modern Xcode — you may not see this file

  6. In Xcode 13+, the default for new projects is generated Info.plist — you don't see the file in the navigator; instead you edit individual keys via the target's Build Settings (INFOPLIST_KEY_* entries) or the Info tab of the target editor. Xcode generates the actual Info.plist at build time. That's why the book line says "may be absent if using build settings."

  7. You can still opt back into a manual Info.plist file if you prefer working in the plist directly — especially when you have lots of custom keys or you're sharing config across targets.

  8. Two vantage points for editing it

  9. You almost always edit Info.plist from one of two places, and the same key can change behavior depending on which path you took. Knowing which is which keeps you out of trouble.

  10. Vantage point 1: Signing & Capabilities tab (the blue + button)

  11. Click your target in the project navigator, then the Signing & Capabilities tab at the top. The + Capability button (the blue plus) opens a list of every Apple capability: iCloud, Push Notifications, HealthKit, Sign in with Apple, Background Modes, App Groups, Keychain Sharing, and so on.

  12. When you toggle a capability ON from this panel, Xcode does three things at once for you:

    • Writes the matching entitlement into the .entitlements file.
    • Inserts the corresponding Info.plist usage-description keys with placeholder text like $(PRODUCT_NAME) needs access to your health data.
    • Registers the capability with your developer-portal App ID so the provisioning profile carries it.

  14. This is the "managed" path. It's the right starting point for any new capability — Xcode keeps the three pieces (entitlement, plist key, profile) in sync.

  15. Vantage point 2: editing the Info.plist file directly

  16. Open Info.plist in the navigator (or the target's Info tab if you're using the auto-generated path) and you get a key-value table where you can add or change anything. This is where you:

    • Replace placeholder usage-description strings with the real reader-facing text. App Review reads these strings. A submission with the literal text $(PRODUCT_NAME) needs access to your health data still in place will get rejected.
    • Add custom URL schemes (CFBundleURLTypes) — not exposed via Capabilities.
    • Add document types (CFBundleDocumentTypes) — ditto.
    • Edit App Transport Security exceptions (NSAppTransportSecurity) — ditto.
    • Add custom keys that third-party libraries read (Mapbox, GoogleSignIn, Firebase configuration, etc.).

  18. Common pitfalls (the cautions worth knowing)

  19. Placeholder strings ship to App Review unless you edit them. The Capabilities tab gives you "$(PRODUCT_NAME) wants permission" placeholders. Real users see those text strings in iOS permission prompts; reviewers reject submissions where the placeholder still has a literal $(...) in it. Always switch to direct-edit and write a plain-language sentence explaining why your app needs the permission.
  20. Entitlement and plist key can drift. If you toggle a capability ON via Capabilities, then later open the entitlements file and remove a key by hand (or vice versa), the two stop matching. Code-signing gets strict about this — an entitlement claimed without a provisioning-profile match crashes the app at launch. Re-toggle the capability via the Capabilities tab to re-sync.
  21. Auto-generated Info.plist makes the file invisible. Modern Xcode (13+) defaults to "generated Info.plist" — you don't see the file in the navigator. Edit via the target's Info tab or the Build Settings INFOPLIST_KEY_* entries. Don't go looking for a missing file thinking something's broken.
  22. Don't mix manual + generated paths in one target. If both a literal Info.plist file AND INFOPLIST_KEY_* build settings exist, the build picks one and silently ignores the other. Pick one approach per target and stay there.
  23. Some capabilities aren't in the +Capability list. A few specialized entitlements (older Mac sandboxing exceptions, certain enterprise capabilities) require manually adding entries to .entitlements. If a capability you need isn't in the list, search Apple's documentation for the entitlement key string and add it directly.
  24. Build number lives in two possible places. CFBundleVersion in Info.plist OR the CURRENT_PROJECT_VERSION build setting. If both are set, build settings win — the Info.plist value gets ignored at build time. Pick the build settings path for new projects; it plays nicer with CI tools that bump the build number on each upload.

  25. The keys you'll edit most often

  26. Identity & versioning

    • CFBundleIdentifier — your reverse-DNS bundle ID, e.g., com.fluharty.MyApp. Must match the one registered with your Apple Developer account.
    • CFBundleShortVersionString — the marketing version users see (1.0, 1.1, 2.0).
    • CFBundleVersion — the build number; must be unique per uploaded build to App Store Connect (1, 2, 3 or 1.0.1, 1.0.2).
    • CFBundleDisplayName — what shows under the icon on Home Screen if it differs from the executable name.

  28. Privacy usage descriptions — the most common reason to edit Info.plist

  29. Every protected API the user must approve has a usage description string. iOS shows your string in the permission prompt. Apple rejects builds that access these APIs without the matching key. Common ones:

    • NSCameraUsageDescription — "Used to scan documents."
    • NSMicrophoneUsageDescription — "Used to record voice notes."
    • NSPhotoLibraryUsageDescription — "Used to import photos into your journal."
    • NSLocationWhenInUseUsageDescription — "Used to stamp your weather entries with your location."
    • NSContactsUsageDescription — "Used to find friends already on the app."
    • NSCalendarsUsageDescription, NSRemindersUsageDescription, NSHealthShareUsageDescription, NSHealthUpdateUsageDescription, NSSpeechRecognitionUsageDescription, NSBluetoothAlwaysUsageDescription...

  31. Capabilities & behaviors

    • UIRequiredDeviceCapabilities — an array declaring "this app won't install on a device without GPS / haptic engine / Metal / etc." Restricts which devices the App Store offers your app to.
    • UISupportedInterfaceOrientations — portrait, landscape-left, landscape-right, upside-down. Per-device variants exist for iPad.
    • UIApplicationSceneManifest — declares your app supports multiple scenes / multi-window on iPad.
    • UIBackgroundModes — an array of background-work types your app uses (audio, location, fetch, processing, remote-notification).
    • LSApplicationCategoryType — macOS app category (public.app-category.productivity, etc.) for the Mac App Store.

  33. URL schemes & deep links

    • CFBundleURLTypes — declare a custom URL scheme (myapp://) so other apps can deep-link into yours.
    • LSApplicationQueriesSchemes — whitelist the schemes you want to query (UIApplication.canOpenURL only works on schemes you've listed here on iOS 9+).

  35. Watch out for

    • Privacy strings are user-visible. Write them in plain language for the end user, not as TODOs to yourself.
    • Build number must increment. If you re-upload with the same CFBundleVersion, App Store Connect rejects.
    • Auto-generated vs manual is one or the other. Mixing causes Xcode to override your manual edits at build time. Pick a path and stick with it.
