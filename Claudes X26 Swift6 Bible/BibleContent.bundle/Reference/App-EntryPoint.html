Reference/App entry point

  1. App entry point (MyAppApp.swift)

  2. Reference: SwiftUI App lifecycle

  3. What this file is

  4. Every Swift app needs exactly one entry point — the place where the operating system hands control to your code at launch. In SwiftUI, that's a struct marked @main conforming to the App protocol. Xcode names it <TargetName>App.swift; that's why the file is MyAppApp.swift when the target is MyApp. The doubled "App" looks odd but it's deliberate — the struct is MyAppApp so the name doesn't collide with Apple's App protocol.

  5. import SwiftUI

@main
struct MyAppApp: App {
    var body: some Scene {
        WindowGroup {
            ContentView()
        }
    }
}

  6. What @main does

  7. The @main attribute tells the compiler "start here." Behind the scenes it generates a main() function that the OS calls. You can only have one @main in your entire target; multiple breaks the build. If you migrate from an older app that used @UIApplicationMain or a separate main.swift, you'd remove those when you adopt the SwiftUI App protocol.

  8. How it auto-handles iPhone, iPad, Mac, Vision

  9. The body returns a Scene — the unit of windowing on each Apple platform. WindowGroup is the most common scene type and it's smart about each platform automatically:

    • iPhone — one window full-screen. WindowGroup gives you that, no configuration needed.
    • iPad — supports Split View, Slide Over, multiple windows of the same app. WindowGroup makes the app multi-window-capable for free; users can drag the app icon from the dock to spawn a second window of the same app.
    • Mac — multiple windows are the default. WindowGroup gives users Cmd+N to open a new window, with each window holding its own ContentView state.
    • Vision Pro (visionOS) — floating windows in 3D space. WindowGroup is still the right choice; the OS adapts the visual treatment.

  11. You don't write platform branches for any of this. Same code, different platform = different windowing behavior.

  12. Other Scene types

    • DocumentGroup — for document-based apps (Pages, Numbers, your-app-that-edits-files). Each open document gets its own window automatically; the system handles open/save/recents.
    • Settings — macOS only. Defines your app's Settings... menu (Cmd+, on Mac). One per app.
    • MenuBarExtra — macOS only. A menu-bar item with a popover or pull-down menu (no main window required).
    • WindowGroup(id:) — multiple distinct WindowGroups, each routable by ID via openWindow. Handy for "open this in its own window."

  14. You can stack multiple Scenes in your body by using a SceneBuilder result — a Mac app might have a WindowGroup for the main UI plus a Settings scene plus a MenuBarExtra all in the same App.

  15. Scene phases — reacting to background/foreground

  16. Read @Environment(\.scenePhase) inside any view to know whether the scene is .active (visible and interactive), .inactive (visible but not focused), or .background (off-screen). Watch for transitions with .onChange(of: scenePhase) — that's where you save state, pause timers, or refresh data when the user returns.

  17. App delegate when you need it

  18. Some Apple APIs still want a UIKit/AppKit app delegate (push notification handlers, deep-link routing in older patterns, etc.). Bridge with @UIApplicationDelegateAdaptor on iOS or @NSApplicationDelegateAdaptor on Mac:

  19. @main
struct MyAppApp: App {
    @UIApplicationDelegateAdaptor(MyAppDelegate.self) var delegate
    var body: some Scene {
        WindowGroup { ContentView() }
    }
}
