Part III The User Interface/Book 12 Sheets Alerts And Confirmations

  1. Chapter 12: Sheets, Alerts, and Confirmations

  2. Sheets

  3. Basic Sheet

  4. Present a modal sheet with a boolean binding:

  5. @State private var showSettings = false

Button("Settings") { showSettings = true }
    .sheet(isPresented: $showSettings) {
        SettingsView()
    }

  6. The sheet dismisses automatically when showSettings becomes false.

  7. Sheet with an Item

  8. Present a sheet driven by an optional identifiable value:

  9. @State private var selectedProject: Project?

List(projects) { project in
    Button(project.name) {
        selectedProject = project
    }
}
.sheet(item: $selectedProject) { project in
    ProjectDetailView(project: project)
}

  10. The sheet appears when the value is non-nil and dismisses when it becomes nil. This is the preferred pattern when the sheet content depends on which item was tapped.

  11. Watch out: The item type must conform to Identifiable. If it doesn't, add conformance or use the boolean variant.

  12. Dismissing a Sheet from Inside

  13. Use the dismiss environment value:

  14. struct SettingsView: View {
    @Environment(\.dismiss) private var dismiss

    var body: some View {
        NavigationStack {
            Form { /* ... */ }
                .toolbar {
                    Button("Done") { dismiss() }
                }
        }
    }
}

  15. Watch out: Always wrap sheet content in a NavigationStack if you want a toolbar with Done/Cancel buttons. Without it, .toolbar has nowhere to render.

  16. Preventing Dismissal

  17. Stop the user from swiping down to dismiss:

  18. .sheet(isPresented: $showEditor) {
    EditorView()
        .interactiveDismissDisabled(hasUnsavedChanges)
}

  19. Pass a boolean to conditionally prevent dismissal. When true, the swipe-down gesture is disabled and the user must use your explicit dismiss button.

  20. onDismiss Callback

  21. Run code when the sheet closes:

  22. .sheet(isPresented: $showPicker, onDismiss: {
    refreshData()
}) {
    PhotoPicker()
}

  24. Sheet Detents

  25. Control how tall a sheet is on iPhone/iPad.

  26. Built-in Detents

  27. .sheet(isPresented: $showInfo) {
    InfoView()
        .presentationDetents([.medium, .large])
}
    • .medium -- half screen
    • .large -- full height (default)

  29. The user can drag between the detents you provide.

  30. Fixed and Fractional Detents

  31. .presentationDetents([
    .height(200),          // fixed 200pt
    .fraction(0.3),        // 30% of screen
    .medium,
    .large
])

  32. Custom Detents

  33. For dynamic sizing:

  34. struct CompactDetent: CustomPresentationDetent {
    static func height(in context: Context) -> CGFloat? {
        min(context.maxDetentValue, 400)
    }
}

// Usage
.presentationDetents([.custom(CompactDetent.self), .large])

  35. Detent Selection Binding

  36. Track and control which detent is active:

  37. @State private var selectedDetent: PresentationDetent = .medium

.sheet(isPresented: $showPanel) {
    PanelView()
        .presentationDetents([.medium, .large], selection: $selectedDetent)
}

  38. Drag Indicator

  39. .presentationDragIndicator(.visible)   // show the grabber bar
.presentationDragIndicator(.hidden)    // hide it

  40. Background Interaction

  41. Allow the user to interact with content behind the sheet:

  42. .presentationBackgroundInteraction(.enabled(upThrough: .medium))

  43. This lets taps pass through to the parent view when the sheet is at .medium or smaller.

  44. Sheet Background and Corner Radius

  45. .presentationBackground(.ultraThinMaterial)
.presentationCornerRadius(20)

  47. fullScreenCover

  48. A modal that covers the entire screen. No swipe-to-dismiss gesture.

  49. @State private var showOnboarding = false

.fullScreenCover(isPresented: $showOnboarding) {
    OnboardingView()
}

  50. Works identically to .sheet in terms of API (boolean binding, item binding, onDismiss).

  51. Watch out: fullScreenCover has no swipe-to-dismiss. You must provide your own dismiss button. If you forget, the user is stuck.

  52. struct OnboardingView: View {
    @Environment(\.dismiss) private var dismiss

    var body: some View {
        VStack {
            // onboarding content
            Button("Get Started") { dismiss() }
        }
    }
}

  53. fullScreenCover does not support detents. It is always full screen.

  55. Popover

  56. A floating bubble anchored to the source view. On iPad and Mac it appears as an actual popover. On iPhone it falls back to a sheet.

  57. @State private var showTooltip = false

Button("Info") { showTooltip = true }
    .popover(isPresented: $showTooltip) {
        Text("This is additional information about the feature.")
            .font(.body)
            .padding()
    }

  58. Popover with Arrow Edge

  59. .popover(isPresented: $showPopover, arrowEdge: .top) {
    PopoverContent()
}

  60. Arrow edge options: .top, .bottom, .leading, .trailing. The system may override this if there is not enough space.

  61. Sizing a Popover

  62. On iPad/Mac, constrain the popover size:

  63. .popover(isPresented: $showPopover) {
    PopoverContent()
        .frame(width: 300, height: 400)
}

  64. Watch out: On iPhone, popovers become sheets. Do not rely on popover-specific behavior for iPhone layouts.

  66. Alerts

  67. Basic Alert

  68. @State private var showAlert = false

Button("Delete") { showAlert = true }
    .alert("Delete Item?", isPresented: $showAlert) {
        Button("Delete", role: .destructive) { deleteItem() }
        Button("Cancel", role: .cancel) { }
    } message: {
        Text("This action cannot be undone.")
    }

  69. Alert with No Message

  70. .alert("Are you sure?", isPresented: $showAlert) {
    Button("Yes") { proceed() }
    Button("No", role: .cancel) { }
}

  71. If you omit the message: parameter, the alert shows only the title and buttons.

  72. Alert with an Item

  73. @State private var errorToShow: AppError?

.alert("Error", isPresented: Binding(
    get: { errorToShow != nil },
    set: { if !$0 { errorToShow = nil } }
)) {
    Button("OK") { errorToShow = nil }
} message: {
    Text(errorToShow?.localizedDescription ?? "")
}

  74. Or more cleanly, make your error type Identifiable and use the item variant:

  75. struct AppError: Identifiable {
    let id = UUID()
    let message: String
}

@State private var currentError: AppError?

.alert(item: $currentError) { error in
    // This closure does not exist in the standard API.
    // Use the pattern below instead.
}

  76. Practical pattern -- the cleanest alert-with-item approach:

  77. @State private var alertMessage = ""
@State private var showErrorAlert = false

func handleError(_ error: Error) {
    alertMessage = error.localizedDescription
    showErrorAlert = true
}

.alert("Error", isPresented: $showErrorAlert) {
    Button("OK", role: .cancel) { }
} message: {
    Text(alertMessage)
}

  78. Alert with TextField

  79. Alerts can contain text fields:

  80. @State private var showRename = false
@State private var newName = ""

.alert("Rename", isPresented: $showRename) {
    TextField("New name", text: $newName)
    Button("Rename") { rename(to: newName) }
    Button("Cancel", role: .cancel) { }
} message: {
    Text("Enter a new name for this item.")
}

  81. Watch out: Alert text fields are iOS 16+ / macOS 13+.

  82. Button Roles

    • .destructive -- red text, signals danger
    • .cancel -- bold text, always placed in the "safe" position
  84. .alert("Reset?", isPresented: $showReset) {
    Button("Reset Everything", role: .destructive) { reset() }
    Button("Cancel", role: .cancel) { }
}

  85. If you provide no .cancel button, the system adds one automatically.

  87. Confirmation Dialog

  88. An action sheet on iPhone, a popover menu on iPad/Mac. Use for choices where the user picks from options.

  89. @State private var showDialog = false

Button("Share") { showDialog = true }
    .confirmationDialog("Share As", isPresented: $showDialog, titleVisibility: .visible) {
        Button("PDF") { shareAsPDF() }
        Button("Image") { shareAsImage() }
        Button("Text") { shareAsText() }
        Button("Cancel", role: .cancel) { }
    } message: {
        Text("Choose the export format.")
    }

  90. Title Visibility

  91. .confirmationDialog("Title", isPresented: $show, titleVisibility: .visible) { }
.confirmationDialog("Title", isPresented: $show, titleVisibility: .hidden) { }
.confirmationDialog("Title", isPresented: $show, titleVisibility: .automatic) { }

  92. On iPhone, .automatic hides the title. Set .visible if you want users to see it.

  93. Confirmation Dialog with Item

  94. @State private var itemToDelete: Item?

.confirmationDialog("Delete", isPresented: Binding(
    get: { itemToDelete != nil },
    set: { if !$0 { itemToDelete = nil } }
), titleVisibility: .visible) {
    if let item = itemToDelete {
        Button("Delete \(item.name)", role: .destructive) {
            delete(item)
            itemToDelete = nil
        }
    }
    Button("Cancel", role: .cancel) { itemToDelete = nil }
} message: {
    Text("This will permanently remove the item.")
}

  96. Inspector

  97. The .inspector modifier presents a side panel on iPad/Mac and a sheet on iPhone. Ideal for detail/property panels.

  98. @State private var showInspector = false

NavigationStack {
    ContentView()
        .toolbar {
            Button("Inspector", systemImage: "info.circle") {
                showInspector.toggle()
            }
        }
        .inspector(isPresented: $showInspector) {
            InspectorContent()
                .inspectorColumnWidth(min: 200, ideal: 300, max: 400)
        }
}

  99. Inspector Column Width

  100. .inspectorColumnWidth(300)                          // fixed width
.inspectorColumnWidth(min: 200, ideal: 300, max: 400)  // flexible range

  101. Watch out: .inspector was introduced in iOS 17 / macOS 14. On older systems, you need a different approach (NavigationSplitView sidebar, etc.).

  103. Presentation Order and Stacking

  104. Multiple Sheets

  105. You cannot attach two .sheet modifiers to the same view. Each view gets one .sheet, one .fullScreenCover, one .alert, etc.

  106. // WRONG -- only one of these will work
Button("Tap")
    .sheet(isPresented: $showA) { ViewA() }
    .sheet(isPresented: $showB) { ViewB() }

// RIGHT -- attach to different views
VStack {
    Button("Show A") { showA = true }
        .sheet(isPresented: $showA) { ViewA() }

    Button("Show B") { showB = true }
        .sheet(isPresented: $showB) { ViewB() }
}

  107. Or use an enum to drive a single sheet:

  108. enum ActiveSheet: Identifiable {
    case settings, profile, editor

    var id: Self { self }
}

@State private var activeSheet: ActiveSheet?

.sheet(item: $activeSheet) { sheet in
    switch sheet {
    case .settings: SettingsView()
    case .profile: ProfileView()
    case .editor: EditorView()
    }
}

  109. This is the cleanest pattern for views that can present multiple different sheets.

  110. Sheet Over Sheet

  111. You can present a sheet from inside a sheet. The new sheet appears on top:

  112. struct FirstSheet: View {
    @State private var showSecond = false

    var body: some View {
        Button("Show Second") { showSecond = true }
            .sheet(isPresented: $showSecond) {
                SecondSheet()
            }
    }
}

  113. Watch out: Deep sheet stacking gets confusing for users. Limit to two levels at most.

  115. Practical Patterns

  116. Destructive Action Confirmation

  117. struct DeleteButton: View {
    let itemName: String
    let onDelete: () -> Void
    @State private var showConfirm = false

    var body: some View {
        Button("Delete", role: .destructive) {
            showConfirm = true
        }
        .confirmationDialog("Delete \(itemName)?",
                            isPresented: $showConfirm,
                            titleVisibility: .visible) {
            Button("Delete", role: .destructive) { onDelete() }
            Button("Cancel", role: .cancel) { }
        } message: {
            Text("This cannot be undone.")
        }
    }
}

  118. Error Alert Modifier

  119. A reusable pattern for showing errors:

  120. struct ErrorAlert: ViewModifier {
    @Binding var error: Error?

    var isPresented: Binding<Bool> {
        Binding(
            get: { error != nil },
            set: { if !$0 { error = nil } }
        )
    }

    func body(content: Content) -> some View {
        content.alert("Error", isPresented: isPresented) {
            Button("OK", role: .cancel) { }
        } message: {
            Text(error?.localizedDescription ?? "An unknown error occurred.")
        }
    }
}

extension View {
    func errorAlert(_ error: Binding<Error?>) -> some View {
        modifier(ErrorAlert(error: error))
    }
}

// Usage
@State private var lastError: Error?

SomeView()
    .errorAlert($lastError)

  121. Half-Sheet Settings Panel

  122. @State private var showSettings = false

.sheet(isPresented: $showSettings) {
    NavigationStack {
        SettingsForm()
            .navigationTitle("Settings")
            .toolbar {
                ToolbarItem(placement: .confirmationAction) {
                    Button("Done") { showSettings = false }
                }
            }
    }
    .presentationDetents([.medium, .large])
    .presentationDragIndicator(.visible)
    .presentationBackground(.regularMaterial)
}
