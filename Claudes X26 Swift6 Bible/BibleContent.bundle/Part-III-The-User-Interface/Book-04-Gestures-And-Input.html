Part III The User Interface/Book 04 Gestures And Input

  1. Chapter 4: Gestures & Input

  2. Claude's Swift Reference 26 -- Part III: The User Interface

  4. What This Chapter Is About

  5. Books 01 through 03 built up the language, SwiftUI's view model, and the scene/window structure of an Apple app. The app runs; views draw. Now we turn to the other half of an app's life: how the user talks back. Taps, swipes, drags, pinches, keyboard presses, Apple Pencil strokes.

  6. SwiftUI covers this with two layers of API:

    • High-level shortcuts built into the common views -- Button, TextField, Toggle, onTapGesture, onSubmit. Most apps live here and go no deeper.
    • The Gesture protocol -- a composable system for building, combining, and tracking touch gestures when the shortcuts don't cover what you need.

  8. This chapter walks both layers. The goal is that by the end you can handle any ordinary input path a user will hand your app, and know which door to open for the uncommon ones.

  10. The Tap -- The Simplest Input

  11. A tap is a single, quick press-and-release in roughly the same spot. Every tappable element in an Apple app begins with a tap.

  12. onTapGesture

  13. The shortest form. Attach .onTapGesture to any view and hand it a closure.

  14. Text("Hello")
    .onTapGesture {
        print("tapped")
    }

  15. That's it. The view becomes tappable; the closure fires every time.

  16. You can ask for a specific number of taps:

  17. Text("Double-tap me")
    .onTapGesture(count: 2) {
        print("double-tap")
    }

  18. And you can read the tap location on iOS 16 and later:

  19. GeometryReader { proxy in
    Color.blue
        .onTapGesture { location in
            print("tapped at \(location)")
        }
}

  20. The location is in the view's own coordinate space.

  21. Button vs onTapGesture

  22. Button and .onTapGesture both handle taps. They are not interchangeable.

  23. Use Button when the tap triggers an action the user can describe in words: "Save", "Delete", "Next". The system gets you accessibility, focus behavior on Apple TV, keyboard activation, and the platform-correct pressed state for free.

  24. Use .onTapGesture when the tapped thing is a graphic element -- an image card, a color swatch, a shape on a canvas -- where a button's default styling would fight you.

  25. A rough rule: if VoiceOver should read "Button" when it lands on the view, use Button. If it should read "Image" or "Graphic", use .onTapGesture.

  27. The Long Press

  28. A long press fires after the user holds their finger down for at least 0.5 seconds without moving much.

  29. Image(systemName: "star")
    .onLongPressGesture {
        print("long-pressed")
    }

  30. You can customize how long and how far the finger is allowed to drift:

  31. Image(systemName: "star")
    .onLongPressGesture(minimumDuration: 1.0, maximumDistance: 10) {
        print("long-pressed")
    }

  32. Long-press is the gesture for "I want the extra menu." Context menus, share menus, and drag-to-reorder all start from it. SwiftUI's .contextMenu modifier wires the gesture automatically -- you rarely call onLongPressGesture yourself when a context menu is what you want.

  33. Text("Item")
    .contextMenu {
        Button("Rename") { /* ... */ }
        Button("Delete", role: .destructive) { /* ... */ }
    }

  35. Drag

  36. A drag gesture reports the finger's position continuously from the first touch until the lift.

  37. DragGesture

  38. DragGesture is a Gesture type. You hand it to the .gesture modifier and attach closures with .onChanged and .onEnded:

  39. struct DragDot: View {
    @State private var offset: CGSize = .zero

    var body: some View {
        Circle()
            .fill(.blue)
            .frame(width: 80, height: 80)
            .offset(offset)
            .gesture(
                DragGesture()
                    .onChanged { value in
                        offset = value.translation
                    }
                    .onEnded { _ in
                        withAnimation(.spring) { offset = .zero }
                    }
            )
    }
}

  40. value.translation is a CGSize of how far the finger has moved since the gesture started. value.location is where the finger currently is. value.startLocation is where it first touched down.

  41. The example above moves the dot while dragging and snaps it back on release.

  42. @GestureState vs @State

  43. If you write the drag example above with @State for the offset, you have to clear it yourself in .onEnded. Miss a case and the dot sticks. @GestureState fixes that.

  44. struct DragDot: View {
    @GestureState private var drag: CGSize = .zero

    var body: some View {
        Circle()
            .fill(.blue)
            .frame(width: 80, height: 80)
            .offset(drag)
            .gesture(
                DragGesture()
                    .updating($drag) { value, state, _ in
                        state = value.translation
                    }
            )
    }
}

  45. @GestureState automatically resets to its initial value (.zero) as soon as the gesture ends or is cancelled by the system. You never write cleanup code, and you never have a stuck dot.

  46. The rule of thumb: use @GestureState for the in-flight position you only care about while dragging, and @State for any state you want to persist after the gesture ends (like the dot's final resting spot).

  47. A combined example -- a dot that stays where you drop it:

  48. struct StickyDot: View {
    @State private var position: CGSize = .zero
    @GestureState private var drag: CGSize = .zero

    var body: some View {
        Circle()
            .fill(.orange)
            .frame(width: 80, height: 80)
            .offset(x: position.width + drag.width,
                    y: position.height + drag.height)
            .gesture(
                DragGesture()
                    .updating($drag) { value, state, _ in
                        state = value.translation
                    }
                    .onEnded { value in
                        position.width  += value.translation.width
                        position.height += value.translation.height
                    }
            )
    }
}

  49. The drag value feeds the live offset; position remembers where the dot settled.

  51. Pinch and Rotate

  52. MagnifyGesture and RotateGesture work the same way as DragGesture -- both are Gesture types you attach with .gesture and observe with .onChanged / .onEnded / .updating.

  53. struct Zoomer: View {
    @State private var scale: CGFloat = 1.0
    @GestureState private var livePinch: CGFloat = 1.0

    var body: some View {
        Image(systemName: "photo.artframe")
            .resizable()
            .frame(width: 200, height: 200)
            .scaleEffect(scale * livePinch)
            .gesture(
                MagnifyGesture()
                    .updating($livePinch) { value, state, _ in
                        state = value.magnification
                    }
                    .onEnded { value in
                        scale *= value.magnification
                    }
            )
    }
}

  54. value.magnification is the ratio between the current two-finger distance and the distance at the gesture's start. It begins at 1.0, grows as fingers spread, shrinks as they pinch.

  55. RotateGesture gives you a value.rotation as an Angle:

  56. .gesture(
    RotateGesture()
        .onChanged { value in
            rotation = value.rotation
        }
)

  57. Pinch and rotate are almost always used together on a photo or map view. SwiftUI lets you combine them.

  59. Combining Gestures

  60. Three composition operators let you describe how two gestures relate:

    • .simultaneously(with:) -- both gestures track at the same time
    • .sequenced(before:) -- the second gesture only starts after the first ends
    • .exclusively(before:) -- exactly one of the two recognizes; the other is dropped

  62. Simultaneous -- Pinch and Rotate Together

  63. struct PhotoEditor: View {
    @State private var scale: CGFloat = 1.0
    @State private var rotation: Angle = .zero

    var body: some View {
        Image("kitten")
            .resizable()
            .scaledToFit()
            .scaleEffect(scale)
            .rotationEffect(rotation)
            .gesture(
                MagnifyGesture()
                    .simultaneously(with: RotateGesture())
                    .onChanged { value in
                        if let m = value.first?.magnification { scale = m }
                        if let r = value.second?.rotation    { rotation = r }
                    }
            )
    }
}

  64. When you combine gestures with .simultaneously, the closure receives a tuple-shaped value with .first and .second. Each side is optional because either gesture can be absent at any given instant.

  65. Sequenced -- Long Press Then Drag

  66. A long-press followed by a drag is the native "grab and move" gesture on iOS.

  67. struct GrabAndMove: View {
    @State private var offset: CGSize = .zero
    @State private var isLifted = false

    var body: some View {
        let press = LongPressGesture(minimumDuration: 0.3)
        let drag  = DragGesture()

        return RoundedRectangle(cornerRadius: 16)
            .fill(isLifted ? .yellow : .blue)
            .frame(width: 120, height: 80)
            .offset(offset)
            .gesture(
                press.sequenced(before: drag)
                    .onChanged { value in
                        switch value {
                        case .first(true):
                            isLifted = true
                        case .second(true, let dragValue?):
                            offset = dragValue.translation
                        default: break
                        }
                    }
                    .onEnded { _ in
                        withAnimation(.spring) { offset = .zero }
                        isLifted = false
                    }
            )
    }
}

  68. The sequenced value is itself an enum: .first(...) during the long press, .second(..., dragValue) during the drag that follows.

  69. Exclusively -- Pick One

  70. Use .exclusively(before:) when two gestures could both match and you want only one to win, in a specific priority.

  71. let doubleTap = TapGesture(count: 2).onEnded { /* ... */ }
let singleTap = TapGesture(count: 1).onEnded { /* ... */ }

Text("Tap me")
    .gesture(doubleTap.exclusively(before: singleTap))

  72. Without .exclusively, a single-tap would fire inside every double-tap.

  74. Focus and the Keyboard

  75. SwiftUI's text inputs (TextField, SecureField, TextEditor) handle keystrokes for you. What you usually need to manage is:

    1. Which field has the keyboard right now?
    2. What happens when the user presses Return or Tab?
    3. What keys should trigger app-level actions, regardless of which field is active?

  77. @FocusState -- Who Has the Keyboard

  78. @FocusState binds a property to "which field is focused." You read it to tell which field is active; you write it to move focus programmatically.

  79. struct LoginForm: View {
    enum Field: Hashable { case username, password }

    @State private var username = ""
    @State private var password = ""
    @FocusState private var focus: Field?

    var body: some View {
        VStack {
            TextField("User", text: $username)
                .focused($focus, equals: .username)
                .submitLabel(.next)
                .onSubmit { focus = .password }

            SecureField("Password", text: $password)
                .focused($focus, equals: .password)
                .submitLabel(.go)
                .onSubmit { /* sign in */ }

            Button("Sign In") { /* sign in */ }
        }
        .padding()
        .onAppear { focus = .username }
    }
}

  80. @FocusState can be Bool (for a single-field form) or any Hashable enum (for multiple fields). Each TextField calls .focused($focus, equals: someCase) to claim its identity.

  81. .submitLabel(.next) sets the return key's label. .onSubmit fires when the user presses return.

  82. Keyboard Shortcuts

  83. For Mac apps and iPad apps connected to a hardware keyboard, .keyboardShortcut makes a Button respond to a key combo:

  84. Button("Save") { save() }
    .keyboardShortcut("s", modifiers: .command)

Button("Close") { close() }
    .keyboardShortcut(.cancelAction) // Escape

  85. The second form uses a role instead of a specific key, so the system picks the right key for the current platform.

  86. Shortcuts defined on buttons in a Menu appear as badges in the menu, just like a native app.

  87. onKeyPress -- Raw Keys

  88. When you need to respond to individual keys outside of a text field -- arrow keys in a game, space bar for play/pause -- use .onKeyPress:

  89. struct CursorBox: View {
    @State private var position: CGPoint = .init(x: 100, y: 100)

    var body: some View {
        Circle()
            .fill(.red)
            .frame(width: 30, height: 30)
            .position(position)
            .focusable()
            .onKeyPress(.leftArrow)  { position.x -= 10; return .handled }
            .onKeyPress(.rightArrow) { position.x += 10; return .handled }
            .onKeyPress(.upArrow)    { position.y -= 10; return .handled }
            .onKeyPress(.downArrow)  { position.y += 10; return .handled }
    }
}

  90. .onKeyPress requires the view to be focusable. Return .handled if the view consumed the key, .ignored to let it bubble up.

  92. Apple Pencil and iPad

  93. On iPad, Apple Pencil acts as a pointer device. Every drag gesture you have already written works with the Pencil automatically. Three Pencil-specific paths are worth knowing.

  94. Scribble

  95. Scribble turns Pencil handwriting inside a TextField into typed text. It works with no code -- every standard text input supports it as soon as the user enables Scribble in Settings. You get it for free.

  96. PencilKit

  97. PKCanvasView is UIKit, but bridges cleanly into SwiftUI for apps that want a drawing surface:

  98. import PencilKit

struct Sketchpad: UIViewRepresentable {
    @Binding var drawing: PKDrawing

    func makeUIView(context: Context) -> PKCanvasView {
        let v = PKCanvasView()
        v.drawingPolicy = .anyInput
        v.delegate = context.coordinator
        v.drawing = drawing
        return v
    }

    func updateUIView(_ v: PKCanvasView, context: Context) {
        v.drawing = drawing
    }

    func makeCoordinator() -> Coordinator { Coordinator(self) }

    final class Coordinator: NSObject, PKCanvasViewDelegate {
        let parent: Sketchpad
        init(_ parent: Sketchpad) { self.parent = parent }
        func canvasViewDrawingDidChange(_ canvas: PKCanvasView) {
            parent.drawing = canvas.drawing
        }
    }
}

  99. Drop a PKToolPicker in front and you have a real sketching surface with pen, pencil, eraser, and color picker.

  100. Pencil Hover and Squeeze

  101. Apple Pencil Pro adds hover (a preview before touching) and squeeze (a side-barrel gesture). Both reach SwiftUI through onPencilSqueeze and onContinuousHover on iOS 17.5 and later. Reach for these when the app benefits from a gesture the finger can't produce.

  103. Hit Targets and Accessibility

  104. A touch is about 44 points wide. Interactive views should be at least 44x44 so users can actually hit them. Apple's Human Interface Guidelines publish this as the recommended minimum.

  105. If your graphic is smaller, expand its tap area with .contentShape:

  106. Image(systemName: "heart")
    .font(.system(size: 16))
    .frame(width: 44, height: 44)
    .contentShape(Rectangle())
    .onTapGesture { /* ... */ }

  107. .contentShape tells SwiftUI which area counts for hit-testing, separate from the visible shape. The heart draws at 16-point but responds to taps anywhere in the 44x44 frame.

  108. Every interactive view should also carry an accessibility label:

  109. Image(systemName: "heart.fill")
    .onTapGesture { toggleFavorite() }
    .accessibilityLabel("Favorite")
    .accessibilityAddTraits(.isButton)

  110. VoiceOver now announces the view as a button named "Favorite." Without the traits and label, it reads "heart.fill, image" -- useless.
