  1. Feedback & Errata

  2. This book is a living reference. If something reads wrong, confuses you, or turns out to have drifted out of date, please flag it — corrections are welcome at any scale, from a single typo to a whole-chapter rewrite suggestion.

  3. Three channels, pick the right one

  4. 1. Errata & bugs — GitHub Issues

  5. Use Issues when something is clearly wrong and needs a fix: a broken link, code that doesn’t compile, a factual error, a stale API reference.

    • Open an issue

  7. 2. Questions & reader-to-reader help — GitHub Discussions

  8. Use Discussions for open-ended questions, show-and-tell of what you’re building from the book, or to help another reader who’s stuck. This is where the community talks to itself.

    • Browse or post a discussion

  10. 3. Private feedback — email

  11. For anything you’d rather not post publicly — a correction you want to flag quietly, a question about the book’s direction, a suggestion you’d like to run by Michael directly — email works. The reader app’s Info menu includes a Send Feedback action that pre-fills the subject line with whichever page you’re looking at.

    • michael.fluharty@mac.com

  13. What helps

  14. When you report something, knowing these makes the fix faster:

    • The exact page (the header at the top of every page shows the path)
    • The line number (every paragraph in this book is line-numbered)
    • What you expected vs. what you saw
    • If it’s a code sample: the Xcode version and the platform you ran it on
