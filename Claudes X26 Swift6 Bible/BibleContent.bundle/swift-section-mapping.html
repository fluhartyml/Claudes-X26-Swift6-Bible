  101. MarkNamePrimary uses in Swift
    :ColonType annotation (x: Int), dictionary pair, inheritance (class A: B), labeled statement, ternary
    ;SemicolonOptional statement terminator; required only to put multiple statements on one line
    ,CommaSeparator in argument lists, tuple elements, collection literals
    .DotMember access, enum case shorthand (.red), decimal point
    ( )ParenthesesGrouping expressions, tuple literals, function parameters and calls
    [ ]Square bracketsArray literals, dictionary literals, subscript access
    { }BracesBlocks, closures, type bodies, get/set blocks
    < >Angle bracketsGeneric type parameters (Array<Int>), generic constraints
    ->ArrowFunction return type (func foo() -> Int), closure return type
    =>Fat arrowNot a Swift operator — mentioned so readers know it isn't. Use ->.
    _UnderscoreWildcard in patterns, ignored parameter label, unused binding

    `name`BackticksEscape a reserved word so it can be used as an identifier

  103. 20. Relationship Clusters (cross-references)

  104. The way readers will actually use the lexicon — not one word at a time, but families. Every entry in these clusters cross-links to the others.

  105. 20.1 Optional family

    Optional · nil · ? · ! · ?? · ?. · if let · guard let · try? · as?

    The single most important cluster for new readers. Optionals are Swift's defining feature.

  106. 20.2 Value-type family

    struct · enum · let · var · mutating · Copy-on-Write (behavior)

  118. 21. Open Mapping Questions

  119. 21.1 — SwiftUI wrappers in the Swift lexicon, yes or no?

    If yes, the S Book gets @State, @StateObject, @SceneStorage, and the E Book gets @Environment, @EnvironmentObject. If no, the lexicon stays Swift-language-pure and SwiftUI gets its own mini-lexicon inside Book 2 (SwiftUI Views).

  120. 21.2 — Operators: their own Book, an appendix, or sprinkled into letter Books?

    Operators aren't alphabetizable. Three shapes: (a) one “Book of Operators” that sits after Z; (b) an appendix at the end of the Swift section; (c) each family cross-linked into the alphabetized Books where the controlling keyword lives (e.g. the Optional operators sit in Book O next to Optional).

  121. 21.3 — Punctuation: same question.

    Probably wants the same answer as operators, whatever that ends up being.

  122. 21.4 — Thin letters (Q, X, Y, Z): keep the empty Books for the encyclopedia feel, or collapse?

    Real encyclopedias ship W-X-Y-Z as one slim volume. We could mirror that.

  123. 21.5 — Concurrency types with capital first letters (Task, TaskGroup, MainActor): alphabetize with the other types, or group them?
  124. 21.6 — Where does the Swift section sit in the overall Bible?

    Options: (a) new Part I before current Part I; (b) companion volume / Volume Two of the Bible; (c) back-of-book reference section.

  125. 21.7 — Entry format: what fields per entry?

    Candidate fields: headword, category tag, plain-English definition, tiny example, related words, double-citation sources. To be decided once the above are settled.

  126. 21.8 — Style-of-truth for concept words that aren’t actually Swift keywords.

    Words like closure, generic, property wrapper, protocol-oriented programming, Copy-on-Write are concepts, not tokens. They probably earn entries anyway — readers will look them up — but they need a different entry shape than a keyword.
